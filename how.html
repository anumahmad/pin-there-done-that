The Idea

Ever since my first year of college, I've been consciously collecting enamel pins. Enamel pins are a very specific kind of collectible. They're small, easy to display, and represent a moment of intention for the artist– each enamel pins requires a custom mold, which means the design is deliberate and meaningful. Over the past few years my collection has grown — over 100 at this point, spanning every corner of who I am: my city, my politics, my obsession with bubble tea, the anime I grew up watching, the books that shaped me. For a long time they just lived on my backpack, then on a poster, which felt like a waste. Every pin has a story. This project was about sharing that story with others.

I wanted to build something that felt less like a portfolio piece and more like a walk through me. The idea of a graph database was something I had been toying with for awhile given the interconnected nature of this project.

Built With Claude Code

This project was built almost entirely through a conversation with Claude Code, Anthropic's AI coding tool. I came in with the idea and the pins. Claude helped me think through the architecture, write every line of code, and work through decisions in real time — kind of like pair programming, except one half of the pair knows every JavaScript API by heart. In college, I had a lot of experience pair programming so it was actually quite natural to take it this step further.

Before a single line of code was written, I detailed what the experience should feel like, what data mattered, how the graph should behave. That conversation became a full project spec document that guided everything after it. You can see the full spec in the repo!

I've been coding since 2018, but this was my first time building something almost entirely through prompting. The experience taught me a lot about how to think out loud, how to be specific about what you want, and what it means to collaborate with a tool that's genuinely capable.

How the Graph Works

There's no server, no real database, and no backend. The "graph database" is a single JSON file sitting in the repo — one object per pin, with fields for name, category, year, how I got it, whether it's a favorite, and more. When you open the page, JavaScript reads that file and computes the graph in your browser.

The edges are the interesting part. Pins connect to their category hubs (the colored hexagons) via category edges. Pins that came from the same trip connect to each other via a dashed silver line. The visual result — clusters of related pins pulling toward shared centers — emerges naturally from the data, not from any manual layout decisions.

Pin data is managed locally through a set of Python scripts: a CSV importer for adding pins in bulk, an interactive CLI for adding one at a time, and an edit tool for updating fields. When the JSON changes, I push to GitHub and the live site updates immediately.

Tech Stack

  • Hosting GitHub Pages — free, static, no server required
  • Graph rendering Cytoscape.js — handles the force-directed layout, node images, and all interactions
  • Data pins.json — a flat JSON array; the entire "database" is ~10KB
  • Frontend Vanilla HTML, CSS, and JavaScript — no framework, no bundler, no build step
  • Data tooling Python 3 scripts (local only) — import from CSV, add a pin, edit a pin
